# requirement-analysis skill

## 流程概览

```text
Step 1: 通读招标文件（建立全局认知）
Step 2: 提取关键信息（招标编号/项目名称/限价/时间节点……）
Step 3: 构建点对点评分台账（评分细则 → 可应答的点）
Step 4: 梳理废标条款清单（否决项红线）
Step 5: 整理技术偏离表（规格条目 + 偏离判定口径）
Step 6: 生成投标书大纲（对齐评分点）
Step 7: 产出招标需求分析报告（综合 + 重点关注）
```

## Step 1: 通读

逐文件读取 `tender/` 下原文。超长文件分段读，边读边记录：章节结构、出现"★/▲"等否决/重要标记的位置、评分办法所在章节、技术规格所在章节。

> 招标文件常见结构：投标须知 → 评标办法 → 合同条款 → 技术规格/需求 → 投标文件格式。重点锁定"评标办法"与"技术需求/规格"。

## Step 2: 提取关键信息 → `analysis/requirement-analysis.md` 顶部

| 字段 | 说明 |
|------|------|
| 招标编号 | 原文出处（页/条） |
| 项目名称 | 全称 |
| 采购人 / 招标代理 | 名称 |
| 预算 / 最高限价 | 金额，超限即废标需标注 |
| 标段划分 | 多标段逐一列出 |
| 投标截止时间 / 开标时间 | 精确到分 |
| 投标保证金 | 金额、形式、截止 |
| 资格要求 | 资质、业绩、人员等门槛 |
| 关键否决标记 | ★/▲ 等实质性条款位置 |

缺失项标注"原文未明确"。

## Step 3: 点对点评分台账 → `analysis/scoring-checklist.md`

把"评标办法"中的每条评分细则拆成**可应答的最小评分点**，建立台账：

```markdown
| 编号 | 评分项 | 评分点（细则） | 分值 | 应答要求 | 承接章节 | 应答状态 |
|------|--------|---------------|------|---------|---------|---------|
| 3.1 | 技术方案 | 总体架构合理性 | 8 | 提供总体架构图+论证 | 第3章 技术方案 | 待应答 |
| 3.2 | 技术方案 | 网络安全设计 | 6 | 提供安全域划分+措施 | 第3章 | 待应答 |
| 4.1 | 服务能力 | SLA 承诺 | 5 | 提供可量化 SLA 指标 | 第4章 服务方案 | 待应答 |
| ... | ... | ... | ... | ... | ... | ... |
| 合计 | | | 100 | | | |
```

**铁律：**
- 每个评分点都要有分值、应答要求、承接章节
- 分值合计必须等于招标评分总分，不等则在台账末尾标注差异原因
- 资格性/符合性条款（即使 0 分）也要列入并标注"资格项/否决项"
- 区分"客观分"（明确量化）与"主观分"（评委裁量），主观分标注提分关键

## Step 4: 废标条款清单 → `analysis/disqualification.md`

逐条列出触发废标/否决投标的情形，标注红线等级：

```markdown
| 编号 | 触发情形 | 原文出处 | 红线等级 | 规避要点 |
|------|---------|---------|---------|---------|
| D1 | 投标报价超最高限价 | 投标须知 2.3 | 致命 | 报价 ≤ 限价 |
| D2 | 未按格式提供法人授权书 | 须知 5.1 | 致命 | 附授权书并签字盖章 |
| D3 | ★条款技术指标负偏离 | 技术需求 6.2 | 致命 | ★项必须无偏离或正偏离 |
| D4 | 投标有效期不足 | 须知 4.2 | 致命 | 有效期 ≥ 要求天数 |
| ... | ... | ... | ... | ... |
```

红线等级：致命（直接废标）/ 高危（极可能废标）。**宁可多列不可漏列。**

## Step 5: 技术偏离表 → `analysis/deviation-table.md`

把技术规格逐条转为待应答偏离表（空应答列，留给 writer 填）：

```markdown
| 序号 | 招标技术要求 | 是否★项 | 投标响应 | 偏离情况 | 偏离说明 |
|------|-------------|--------|---------|---------|---------|
| 1 | CPU ≥ 16 核 | ★ | 待应答 | 待判定 | |
| 2 | 内存 ≥ 64GB | | 待应答 | 待判定 | |
| ... | ... | ... | ... | ... | ... |
```

偏离判定口径：正偏离（优于要求）/ 无偏离（符合）/ 负偏离（低于要求，★项负偏离=废标）。

## Step 6: 投标书大纲 → `analysis/outline.md`

按招标"投标文件组成/格式"要求 + 评分点承接，生成章节目录。每章标注承接的评分点编号：

```markdown
# 投标书大纲

1. 投标函及投标函附录
2. 法定代表人身份证明 / 授权委托书
3. 技术方案
   3.1 总体技术架构【承接 3.1, 3.2】
   3.2 ……
4. 服务方案 / 供货方案 / 施工组织设计（按标书类型）
5. 项目管理与团队
6. 商务部分
7. 资质与业绩证明
...
```

确保**评分台账每个评分点都能在大纲找到承接章节**（双向对齐）。

## Step 7: 招标需求分析报告 → `analysis/requirement-analysis.md`

在关键信息基础上补充：
- **投标可行性**：资格门槛是否满足（标注需作者确认项）
- **得分策略**：客观分必拿项、主观分发力点、可放弃的低性价比项
- **重点关注/风险**：易废标点、易漏项、模糊条款（建议澄清）
- **技术要求摘要**：交给 architect 的核心技术约束

## 验证（Definition of Done）

- [ ] 五份产出齐全（requirement-analysis / scoring-checklist / disqualification / deviation-table / outline）
- [ ] 评分点分值合计 = 招标评分总分（或标注差异）
- [ ] 废标清单覆盖全部否决项，每条有原文出处
- [ ] 偏离表逐条列出技术规格，★项已标记
- [ ] 大纲与评分台账双向对齐
- [ ] 模糊/缺失条款均显式标注，无臆造
