# requirement-analysis skill

> **范围红线：只解析技术。** 信息来源仅限招标文件的 **评分表-技术部分**、**实质性要求-技术产品部分**、**技术任务书 / 采购需求技术部分**。商务、价格、付款、资质业绩一律不纳入。

## 流程概览

```text
Step 1: 定位技术部分（评分表-技术部分 / 实质性要求-技术产品部分 / 技术任务书-采购需求技术部分）
Step 2: 提取技术概况（招标编号/项目名/技术方案类型/技术任务书位置）
Step 3: 构建技术部分评分索引表（评分表-技术部分 → 可应答的技术点，含承接章节索引）
Step 4: 梳理实质性技术否决项清单（实质性要求-技术产品部分）
Step 5: 整理技术偏离表（技术规格条目 + 偏离判定口径）
Step 6: 生成技术方案大纲（对齐技术评分点）
Step 7: 产出技术需求分析报告（综合 + 技术重点）
```

> 本 SOP 独立产出两张关键表格：**技术部分评分索引表**（Step 3）与**技术偏离表**（Step 5）。

## Step 1: 定位技术部分

逐文件读取 `tender/` 下原文，**只锁定三处技术来源**：
- **评分表-技术部分**（评标办法中技术分相关条目，剔除价格分/商务分）
- **实质性要求-技术产品部分**（实质性/否决性条款中与技术产品相关的）
- **技术任务书 / 采购需求技术部分**（功能/性能/接口/安全等技术要求）

> 招标文件常见结构：投标须知 → 评标办法 → 合同条款 → 技术需求/技术任务书 → 投标文件格式。本系统只取"评标办法-技术分"与"技术需求/技术任务书"，其余跳过。

## Step 2: 技术概况 → `analysis/requirement-analysis.md` 顶部

| 字段 | 说明 |
|------|------|
| 招标编号 | 原文出处 |
| 项目名称 | 全称 |
| 技术方案类型 | 软件开发/硬件开发/系统集成/服务（据采购需求判断） |
| 技术任务书位置 | 章节/页 |
| 标段（技术相关） | 多标段逐一列出技术范围 |

> 不记录预算/限价/保证金/付款等商务字段。

## Step 3: 技术部分评分索引表 → `analysis/scoring-checklist.md`

把"评分表-技术部分"中每条技术评分细则拆成**可应答的最小技术评分点**，建立**技术部分评分索引表**（贯穿全流程：规划→撰写回填→随技术方案交付）：

```markdown
| 编号 | 技术评分项 | 技术评分点（细则） | 分值 | 应答要求 | 承接章节 | 应答位置 | 应答状态 |
|------|-----------|-------------------|------|---------|---------|---------|---------|
| 2.1 | 总体技术方案 | 架构合理性 | 8 | 提供架构图+技术论证 | 第2章 | 待回填 | 待应答 |
| 3.1 | 详细设计 | 接口/数据设计 | 6 | 提供接口规范+数据模型 | 第3章 | 待回填 | 待应答 |
| ... | ... | ... | ... | ... | ... | ... | ... |
| 技术分合计 | | | 60 | | | | |
```

**铁律：**
- 只列技术评分点（不含价格分、商务分）
- 每个技术评分点都要有分值、应答要求、承接章节（= 评分点→章节索引）
- "应答位置/应答状态"两列由 updater 在归档时回填（依据 writer 正文的应答标记），形成点对点应答对照索引
- 分值合计 = 评分表技术部分总分，不等则标注差异原因
- 区分"客观技术分"与"主观技术分"，主观分标注提分关键
- 本表终态在 compose 阶段作为附表随 output/technical-proposal.md 交付

## Step 4: 实质性技术否决项清单 → `analysis/disqualification.md`

逐条列出**实质性要求-技术产品部分**中触发否决（废标）的技术情形：

```markdown
| 编号 | 触发情形（技术） | 原文出处 | 红线等级 | 规避要点 |
|------|----------------|---------|---------|---------|
| D1 | 实质性★技术指标负偏离 | 技术需求 6.2 | 致命 | ★项无偏离/正偏离 |
| D2 | 未响应强制技术标准/认证 | 实质性要求 3.1 | 致命 | 逐条响应+附证明 |
| D3 | 必须提供的技术文件缺失 | 实质性要求 3.4 | 致命 | 补齐技术文件 |
| ... | ... | ... | ... | ... |
```

红线等级：致命（直接否决）/ 高危。**宁可多列不可漏列。** 仅限技术产品类否决项，不含商务否决（如保证金、报价超限）。

## Step 5: 技术偏离表 → `analysis/deviation-table.md`

把技术规格逐条转为待应答偏离表（空应答列，留给 writer 填）：

```markdown
| 序号 | 招标技术要求 | 是否实质性★项 | 投标响应 | 偏离情况 | 偏离说明 |
|------|-------------|-------------|---------|---------|---------|
| 1 | CPU ≥ 16 核 | ★ | 待应答 | 待判定 | |
| 2 | 接口支持 RESTful | | 待应答 | 待判定 | |
| ... | ... | ... | ... | ... | ... |
```

偏离口径：正偏离（优于）/ 无偏离（符合）/ 负偏离（低于；实质性★项负偏离=否决）。

## Step 6: 技术方案大纲 → `analysis/outline.md`

按技术任务书结构 + 技术评分点承接，生成技术方案章节目录。每章标注承接的技术评分点编号：

```markdown
# 技术方案大纲

1. 项目技术理解与需求分析【承接 1.x】
2. 总体技术方案与技术架构【承接 2.1, 2.2】
3. 详细技术设计【承接 3.x】
4. 非功能/质量技术设计（性能/安全/可靠性）【承接 4.x】
5. 实施技术方案（部署/迁移/测试，技术侧）【承接 5.x】
6. 技术保障措施【承接 6.x】
7. 技术偏离表
```

确保**技术评分台账每个评分点都能在大纲找到承接章节**（双向对齐）。不规划投标函/商务/资质章节。

## Step 7: 技术需求分析报告 → `analysis/requirement-analysis.md`

格式见 requirement-analysis-spec.md。补充：技术评分办法概览、实质性技术要求、技术任务书要点、技术得分策略、技术风险与关注、交付 architect 的技术约束。

## 验证（Definition of Done）

- [ ] 五份产出齐全（其中独立成表：技术部分评分索引表 scoring-checklist.md、技术偏离表 deviation-table.md）
- [ ] 技术部分评分索引表分值合计 = 评分表技术部分总分（或标注差异）
- [ ] 否决项清单覆盖实质性要求-技术产品部分全部技术否决情形
- [ ] 偏离表逐条列出技术规格，实质性★项已标记
- [ ] 大纲与技术评分台账双向对齐
- [ ] 未混入商务/价格/资质内容；模糊/缺失技术条款显式标注
