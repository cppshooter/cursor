# section-writing skill

> **范围红线：只写技术方案。** 不写商务、报价、付款、资质业绩、投标函等非技术内容。

## 流程概览

```text
Step 1: 准备（读章纲 + 本章技术评分点 + 检索素材 + 否决红线 + 写作记忆）
Step 2: 按技术方案类型选择章节模板与侧重
Step 3: 点对点技术应答撰写（每个技术评分点一段，打应答标记）
Step 4: 类型专属产物（硬件/集成→技术偏离表 / 软件→技术设计 / 服务→技术能力指标）
Step 5: 实质性技术否决规避自检
Step 6: AI 味 + 质量自检
```

## Step 1: 准备

读取：
- `chapters/ch-{N}-{slug}.md`（技术章纲：本章范围、承接技术评分点）
- `analysis/scoring-checklist.md`（本章承接的技术评分点 + 应答要求）
- `.agent/task/retrieval-result.md`（检索员提供的可引用技术素材）
- `analysis/disqualification.md`（实质性技术否决红线）
- `analysis/deviation-table.md`（硬件/集成类应答依据）
- `.claude/memory/writing-memory.md`（作者历轮反馈）
- 重写时：`write-order.md` 中 expert 的失分项与改进建议

## Step 2: 按类型选模板与侧重

加载 `.claude/knowledge/bid-type.md`，按类型组织章节：

**软件开发（software）：**
- 需求理解 → 总体技术方案/架构 → 功能模块设计 → 接口/数据库设计 → 关键技术 → 性能/安全/可用设计 → 测试方案 → 部署/迁移技术方案

**硬件开发（hardware）：**
- 需求与指标分解 → 硬件总体设计 → 电路/结构/热设计 → 选型与关键指标 → 技术参数偏离应答 → 可靠性/EMC/环境适应性 → 生产工艺与测试验证

**系统集成（integration）：**
- 需求与集成目标 → 总体集成架构/拓扑 → 软硬件选型清单 → 技术参数偏离应答 → 集成/互联互通/兼容性 → 迁移割接（含回退）→ 联调测试 → 非功能设计

**服务（service）：**
- 服务需求理解 → 服务技术体系/支撑平台 → 技术服务流程（分级响应）→ 可量化技术能力指标 → 技术保障/应急/容灾技术方案 → 持续技术改进

## Step 3: 点对点技术应答撰写

对本章承接的**每一个技术评分点**，写一段明确应答，并打标记：

```markdown
### 2.1 总体技术架构

【应答技术评分点 2.1：总体架构合理性（8分）】
本项目采用四层总体架构……（用技术指标、图、清单论证，引用 architecture/diagrams/topology）

【应答技术评分点 3.2：安全设计（6分）】
安全域划分为……，部署 WAF 与等保三级措施……
```

**应答原则：**
- 一个技术评分点至少一段实质应答，开头用 `【应答技术评分点 X.Y：名称（分值）】` 标记
- 用可核验内容：技术指标、数据、表格、流程、清单、量化承诺
- 引用检索素材时标注 `（参考类似项目：XX，已去敏）`
- 主观技术分评分点：突出技术差异化亮点与论证深度

## Step 4: 类型专属产物

**硬件/集成类——技术参数偏离表**（填回 deviation-table 对应条目，并在正文呈现）：
```markdown
| 序号 | 招标技术要求 | 实质性★ | 投标响应 | 偏离 | 说明 |
|------|-------------|--------|---------|------|------|
| 1 | CPU≥16核 | ★ | 实配 32 核 | 正偏离 | 优于要求一倍 |
```
实质性★项严禁负偏离（触否决）。

**软件类——技术设计产物：** 系统架构图、功能模块图、接口规范表、数据模型、性能/安全设计指标。

**服务类——技术能力指标表（非商务罚则）：**
```markdown
| 服务技术指标 | 承诺值 | 统计口径 | 技术实现 |
|------------|-------|---------|---------|
| 系统可用率 | ≥99.9% | 月度 | 双活+自动切换 |
| 严重故障技术响应 | ≤15分钟 | 工单时间戳 | 监控告警直达 |
```

**软件/集成——架构与流程：** Mermaid 架构图/拓扑图/时序图/功能模块图（见 diagram-spec）。

## Step 5: 实质性技术否决规避自检

逐条对照 `disqualification.md`（实质性技术否决项），确认本章无任何触碰：
- 实质性★项无负偏离
- 强制技术标准/认证已响应
- 必须的技术文件/承诺已满足或占位提示
- 技术参数无低于硬性门槛

触碰红线 → 立即修正。

## Step 6: AI 味 + 质量自检

加载 `.claude/knowledge/anti-ai.md` 与 `.claude/knowledge/section-quality-checklist.md`：

**防套话/AI 味（违规即改）：**
- 删空喊口号："采用先进技术""高度可靠""一流"等无信息量表述
- 疲劳词阈值：见 anti-ai.md
- 不臆造技术参数（缺失用 `【待提供：...】` 占位）
- 不混入商务/价格/资质内容

**质量自检：**
- 本章技术评分点 100% 有 `【应答技术评分点】` 标记
- 类型产物齐全（偏离表/技术设计/技术能力指标）
- 表格优先于冗长段落
- 术语与 analysis/architecture 一致

## 输出

`sections/ch-{N}-{slug}.draft.md`，写完清理 `write-order.md`。

- [ ] 无未标注的臆造内容（缺失均用占位标注）

- [ ] AI 味/套话自检通过
