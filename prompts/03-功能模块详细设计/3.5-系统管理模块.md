## 二、章节定位与篇幅

第 3 章第 5 个一级模块。**目标页数 7~9 页**（含日志、策略、配置三块）。

## 三、需求映射（严格不扩大）

来自《系统需求.md》「系统管理」原文：

- （1）状态监控：具备查询多种硬件工作状态的功能；具备硬件自检的功能。
- （2）策略管理：具备根据策略生成控制参数的功能；具备增加、修改、删除策略库表内容的功能；具备策略导出功能。
- （3）信息管理：具备硬件配置信息管理功能。

并结合性能要求：

> 日志操作种类 ≥3 种（检索、清除、重置筛选等）；显示的系统状态信息 ≥2 种（磁盘容量、CPU 占用率等）。

拆分为：

- **3.5.1 状态监控子模块**（含：硬件工作状态查询、硬件自检、系统状态信息显示 ≥2 种、日志操作 ≥3 种）
- **3.5.2 策略管理子模块**
- **3.5.3 信息管理子模块**

> 日志操作并入"状态监控"子模块，避免新增"日志中心"独立模块导致扩大需求。

不引入"用户/角色/权限/审计中心/单点登录/消息推送"。

## 四、每个子模块固定五小节

### (1) 功能模块描述
- 表格：输入、输出、依赖。
- 1 段话说明子模块覆盖的需求条目。

### (2) 操作步骤（用户操作手册风）

**3.5.1 状态监控** 操作步骤建议（多视图共存）：

- 入口：主菜单 `视图(V) → 状态监控(S)`，或工具栏【状态监控】。
- 中央 `.qt-tabs`（硬件状态 / 自检 / 系统状态 / 日志），分别说明：
  - **硬件状态**：左侧 `.qt-tree` 选硬件 → 右侧 `.qt-group "状态详情"` 显示运行参数（电压、温度、错误码等）。【刷新】按钮、【自动刷新】开关（`.qt-check`）。
  - **自检**：选择自检项（`.qt-check` 多选：通信链路 / 电源链路 / 文件系统 / 数据库 ...）→ 点击【开始自检】→ 进度条 + 结果列表 → 自检报告导出。
  - **系统状态**：固定显示 ≥2 项（**磁盘容量、CPU 占用率**；可增加内存、网络），用 4 个"指标卡" `.qt-group`，每 1Hz 刷新。
  - **日志**：顶部筛选条 `.qt-row`（时间范围、级别下拉、关键字、模块下拉），3 个明确动作按钮：【检索】【清除】【重置筛选】（呼应"≥3 种操作"），结果区 `.qt-table`，底部【导出日志】可选。
  - 快捷键：F5 刷新、Ctrl+F 检索、Ctrl+L 清除（弹 `QDialog` 二次确认）、Ctrl+R 重置筛选。

**3.5.2 策略管理** 操作步骤建议：
- 入口：主菜单 `工具(T) → 策略管理(S)`。
- 中央 `.qt-table` 策略库（列：策略 ID、名称、版本、生效时间、描述、操作）。
- 操作：
  - 【新增】→ 弹 `QDialog`（策略名称、参数模板、关联硬件、生效时间）→ 保存。
  - 选中行 → 【修改】或双击进入编辑。
  - 【删除】→ `QDialog` 二次确认。
  - 【生成控制参数】→ 选策略 → 弹窗显示生成的参数列表 → 可下发给硬件交互模块。
  - 【导出】→ 弹 `QFileDialog` 保存为文件（`*.json` 或 `*.csv`）。

**3.5.3 信息管理** 操作步骤建议：
- 入口：主菜单 `工具(T) → 硬件配置(H)`。
- 中央 `.qt-table` 硬件配置（列：硬件名称、型号、通信方式、地址、波特率/IP 端口、备注、操作）。
- 操作：【新增】【修改】【删除】【保存】【刷新】，与策略管理界面风格一致。

每个子模块紧跟一张 **PlantUML 时序图**（≤5 参与者）。

### (3) 类与算法设计（C++17 + Qt）
- 状态监控：
  - `HardwareStatusQueryService`、`HardwareSelfTestService`（信号 `selfTestStepFinished(step, ok)`）
  - `SystemMetricsProvider`（封装磁盘 / CPU 采集，可调用 Linux `/proc` 或 Qt 工具）
  - `LogQueryService`（提供 `search / clear / resetFilter`，对应日志 ≥3 种操作）
- 策略管理：
  - `StrategyService`、`ControlParamGenerator`、`StrategyExporter`
  - 数据模型 `StrategyEntity`、`StrategyParamEntity`
- 信息管理：
  - `HardwareConfigService`、`HardwareConfigEntity`
- 必选 1 个核心算法（≤30 行 C++）：**硬件自检流程算法**（多步骤检查 → 每步结果记录 → 整体结论与故障定位）或 **策略→控制参数生成算法**（规则匹配 + 模板填充）。

### (4) 用例描述（PlantUML 用例图）
- 参与者：操作员、硬件、操作系统、下游模块。
- 用例（注意覆盖性能要求）：
  - 查询硬件工作状态
  - 硬件自检
  - 显示系统状态信息（≥2 种）
  - 日志检索 / 日志清除 / 日志重置筛选（≥3 种）
  - 根据策略生成控制参数 / 增加 / 修改 / 删除策略 / 策略导出
  - 硬件配置信息管理
- 节点 ≤14；可分组以保持简洁。

### (5) 界面设计（HTML，严格套用附录 B 模板）
- 整个模块为一个主窗口 + `.qt-tabs` 切换三大子模块视图，或给出三段 HTML（统一骨架）。
- 工具栏建议：【刷新】【自动刷新】【自检】【检索日志】【清除日志】【重置筛选】【新增策略】【导出策略】【新增硬件】。
- 状态栏（强制）：**磁盘 X%** / **CPU Y%**（≥2 种系统状态）+ 当前时间 + 数据库 `.qt-led`。
- 中央工作区：
  - 状态监控 Tab 组（硬件 / 自检 / 系统 / 日志）按上述操作步骤布控件。
  - 策略管理 Tab：`.qt-table` + 右侧 `.qt-group "策略详情"`。
  - 信息管理 Tab：`.qt-table` 配置列表 + 右侧 `.qt-group "配置详情"`。
