# 6. 通用质量特性设计方案

本章逐条对齐《系统需求.md》"通用质量特性要求"中的可靠性、维修性、安全性、保障性、测试性与环境适应性六项要求，给出本系统在设计层面的应对措施。措施一律以可执行的工程实践呈现，不引入需求中未提及的额外质量项。

## 6.1 可靠性

系统需求对可靠性提出三方面要求：采用成熟的软件开发技术、采用成熟的软件编程技术；在界面处理模块中对输入数据进行类型与合法性检查；针对异常处理设计异常处理程序。

**技术选型成熟度**。本系统的技术栈整体围绕长期稳定版本展开。开发语言选用 C++17，编译器采用银河麒麟 V10 自带或软件源中的 GCC，工具链稳定。界面与基础设施选用 Qt 5.15 LTS，模块齐全且具备长期维护承诺。数据库选用 SQLite 3.x（主选）或达梦 DM8、人大金仓 KingbaseES（备选），三者均经过国内信创场景的长期验证。构建工具 qmake 与 CMake、单元测试框架 QtTest、静态检查工具 cppcheck 与 clang-tidy，均为业界长期使用的成熟工具。

**编程技术成熟度**。源代码遵循 Google C++ Style 及内部补充约束，使用 RAII 与智能指针进行资源管理，避免裸指针与手工 `new`/`delete`；并发场景使用 Qt 的事件循环、信号槽与 `QThread`，避免直接操作底层线程同步原语；内存与文件资源在异常路径下也能正确释放。

**输入数据校验**。系统在界面处理模块对操作员输入实施强制校验。文本框使用 `QIntValidator`、`QDoubleValidator`、`QRegExpValidator` 等校验器限制输入；下拉框、数字框、日期时间框等控件本身限定取值范围；提交前由界面层进行字段必填、格式与范围检查。服务层接收输入后再进行类型、长度、范围与枚举值的二次校验，避免界面层被绕过的情况下产生异常。

**异常处理**。系统设计分层异常体系：基类 `AppException` 提供错误码、错误消息与上下文；派生类 `BusinessException`、`HardwareException`、`DbException`、`IoException` 分别对应业务、硬件、数据库与文件 IO 场景。关键路径（任务接收、指令下发、结果入库、数据库读写、文件读写）使用 `try/catch` 进行捕获，异常进入统一的异常处理函数，写入 `t_sys_log` 表并以人可读的方式反馈到界面（弹出 `QMessageBox` 或状态栏提示）。对硬件交互通路，额外配合超时、重试与心跳机制，将瞬时故障吸收在重试过程中，避免传播到业务层。

## 6.2 维修性

系统需求对维修性提出三方面要求：软件具备模块化、可变规模与适应未来改变的灵活性，按"高内聚、低耦合"划分模块；编程风格统一，注释行不少于代码行的 30%；交付时提供软件用户手册。

**模块化与可裁剪**。系统按系统需求中的五个一级功能模块划分独立目录与命名空间，每个模块下再按系统需求中的原子能力拆分子模块。模块对外仅暴露抽象基类与必要的数据结构，模块之间通过 Qt 信号槽进行异步通信。通用能力下沉到 common 库，所有模块对 common 单向依赖。这一结构使得新增子模块或替换实现时，影响面被限制在自身目录内。

**编程风格一致**。系统在工程模板阶段提交一份 `.clang-format` 与 `.clang-tidy` 配置，统一缩进、行宽、括号风格与命名规则。开发者在 IDE 中保存文件时自动格式化，提交前通过 `pre-commit` 钩子运行静态检查与注释率统计。代码评审在合入前对风格、可读性、注释充分性进行审查，未通过的修改不予合入。

**注释规范**。注释分为三个层次：文件头注释说明文件用途、作者与版本；模块、类、函数级注释采用 Doxygen 兼容格式，列出用途、参数、返回值、可能抛出的异常；行内注释用于解释非显而易见的实现意图，例如算法选择理由、性能考量、与硬件协议的衔接点。持续集成流水线在编译前自动统计注释率，注释行数与总代码行数的比例不低于 30%。

**软件用户手册**。交付时随软件安装包提供软件用户手册，覆盖系统启动、五大模块的界面元素、菜单与工具栏按钮、典型操作流程、常见问题与故障排查。手册在维护期内随版本更新同步修订。

## 6.3 安全性

系统需求对安全性提出两条要求：输入信息的合法性检查与误操作防护。

**输入信息合法性检查**。本系统在界面层、服务层与接口层分别设置校验关卡，对来自操作员、上层任务软件、硬件以及离线文件的全部输入进行类型、范围、长度、格式与业务合法性校验。界面层使用 Qt 自带的 `QValidator` 系列控件；服务层在接收输入后再进行业务一致性校验；接口层在协议层面对帧头、长度、CRC 进行三层校验，并对解码后的业务字段进行枚举与范围校验。任何一处校验失败均不进入后续业务流程，并将失败原因写入日志。

**误操作防护**。对于可能造成数据丢失或硬件状态改变的危险操作，系统统一采用二次确认机制。具体包括：清除日志、删除策略、清空硬件配置、电源开机、批量删除任务、覆盖式导入策略库等。点击对应按钮后弹出 `QMessageBox::question` 对话框，明确告知操作影响并提示无法撤销，操作员点击"确定"后方执行；点击"取消"或关闭对话框则放弃操作。危险操作执行前后均写入 `t_sys_log`，包含操作员标识、时间、模块与内容，便于事后追溯。

**关于用户与权限**。系统需求未提出用户管理、角色、权限子系统，本方案不引入相关设计。如后续提出，将作为变更需求按 GJB 438C-2021 变更评审流程处理。

## 6.4 保障性

系统需求对保障性提出明确条款：乙方提供给甲方为期 5 年的维护，维护期内软件发生故障或缺陷，乙方负责解决。

本系统的保障性设计围绕"5 年维护期内的稳定可维护"展开。从产品维度，源代码、设计文档、测试材料按规范归档，归档介质满足维护期内随时检索的需要；从过程维度，建立服务台、缺陷库、版本发布流程，确保每一次报障可追溯、每一次修复有依据；从组织维度，乙方在维护期内保持稳定的服务团队，并按甲方要求派人参加系统联调、所检、外场、用户验收。

保障性的具体内容、响应分级、版本管理、文档与培训等条款，在第 9 章售后服务与保障方案中详细给出。本节仅明确：维护期 5 年；维护期内的全部软件故障与缺陷由乙方负责解决。

## 6.5 测试性

系统需求对测试性提出四方面要求：合格性测试在指定的系统上进行；软件测试类型包括功能、性能、边界、安全性、接口；合格性测试依据来源于技术协议，符合率达到 100%，应测试合法边界与非法边界，每一个软件特性必须至少被一个正常用例覆盖；对于不具备条件的测试项，确定测试与验证时机和方法。

**测试环境**。合格性测试在指定的银河麒麟 V10 上位机上进行，硬件配置与现场一致。测试环境的搭建步骤、依赖版本、数据库初始数据随测试方案一同归档。

**测试类型与方法**。系统覆盖五类测试：功能测试按功能用例验证业务正确性；性能测试针对日志、系统状态显示、接口响应等场景在指定数据规模与频率下验证响应时间；边界测试针对每个输入字段的合法上下界与非法值各设计用例；安全性测试覆盖输入合法性校验与危险操作二次确认；接口测试针对控制指令、状态信息与指令执行结果三类报文按字段、长度、CRC、序列号等维度组织用例。

**测试用例覆盖**。合格性测试以技术协议为依据，符合率 100%。每个软件特性至少配置一个正常路径用例；对每一个输入字段，按合法上下界、合法常规值、非法越界值各设计用例。功能、性能、边界、安全性、接口五类测试按模块组织覆盖表，覆盖表随测试方案与测试报告一同提交。

**不具备条件的测试项**。对于硬件不到位、网络不具备的场景，系统提供仿真器与桩进行替代验证。仿真器按统一报文结构模拟硬件的状态信息与执行回执；桩在数据访问层与硬件交互层之间插入，用于注入异常路径。仿真器与桩在测试方案中说明使用范围与验证方法。

| 模块 | 功能 | 性能 | 边界 | 安全性 | 接口 |
|---|---|---|---|---|---|
| 任务管理 | ✓ | ✓ | ✓ | ✓ | ✓ |
| 数据处理 | ✓ | ✓ | ✓ | ✓ | — |
| 硬件交互 | ✓ | ✓ | ✓ | ✓ | ✓ |
| 结果评估 | ✓ | ✓ | ✓ | ✓ | ✓ |
| 系统管理 | ✓ | ✓ | ✓ | ✓ | — |

**单元测试**。单元测试使用 QtTest 框架组织，覆盖业务层与数据访问层。每个 Service 类至少包含正常路径、异常路径与边界路径三类用例；数据访问层针对每张表的增删改查写测试用例，使用 SQLite 内存模式（`:memory:`）作为测试数据库以提升速度。

## 6.6 环境适应性

系统需求对环境适应性的要求为"无要求"。本系统部署于银河麒麟 V10 上位机，运行环境由操作系统与硬件提供，未引入额外的温度、湿度、振动、电磁兼容等环境约束。如后续提出环境适应性要求，按变更需求纳入设计与测试范围。
