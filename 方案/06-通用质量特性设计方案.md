# 6. 通用质量特性设计方案

## 6.1 可靠性

| 需求条目 | 设计措施 |
|---|---|
| 采用成熟的软件开发技术 | C++17 + Qt 5.15 LTS（LGPL）+ SQLite 3.x，均为长期稳定版本；构建工具 qmake / CMake，单元测试 QtTest |
| 采用成熟的软件编程技术 | 遵循 Google C++ Style；使用 RAII、智能指针管理资源；信号槽解耦模块 |
| 界面层输入数据类型与合法性检查 | 控件级使用 `QIntValidator` / `QDoubleValidator` / `QRegExpValidator` 限制输入；服务层对长度、范围、枚举值二次校验 |
| 异常处理 | 分层异常体系：`AppException` 基类，下分 `BusinessException` / `HardwareException` / `DbException`；关键路径捕获并写入 `t_sys_log` |
| 硬件链路可靠性 | 控制指令默认重试 2 次、超时 5 s；硬件心跳 1 Hz，连续丢失 3 次置离线并告警 |

## 6.2 维修性

| 需求条目 | 设计措施 |
|---|---|
| 模块化、可变规模 | 五大功能模块独立目录与命名空间；通用能力（日志、配置、数据库访问、消息编解码）下沉 common 库 |
| "高内聚、低耦合" | 模块间通过抽象基类与 Qt 信号槽通信，避免相互直接依赖；接口变更通过基线评审 |
| 编程风格一致 | 强制 Google C++ Style + 内部补充规范；提交前运行 cppcheck / clang-tidy 静态检查 |
| 注释行 ≥ 总代码行 30% | Doxygen 格式文档化注释 + 行内注释；CI 步骤中自动统计注释率，低于 30% 不允许合入 |
| 提供软件用户手册 | 与软件安装包同步交付，覆盖五大模块界面、菜单与操作步骤 |

## 6.3 安全性

| 需求条目 | 设计措施 |
|---|---|
| 输入信息合法性检查 | 界面层 + 服务层双校验；接口报文按帧头、长度、CRC16 三层校验 |
| 误操作防护 | 危险操作（清除日志、删除策略、清空配置、电源开机、批量删除任务）统一弹出 `QMessageBox::question` 二次确认，操作员明确点击"确定"后方执行 |
| 关键操作可追溯 | 上述操作写入 `t_sys_log`，含操作员标识、时间、模块、内容；状态栏左下角实时显示最近一次关键操作 |

> 注：系统需求未提出用户、角色、权限子系统，本方案不引入相关设计。如后续提出，将作为变更评审项处理。

## 6.4 保障性

| 项 | 内容 |
|---|---|
| 维护期 | 验收合格之日起 5 年 |
| 维护责任 | 维护期内的软件故障与缺陷由乙方负责解决 |
| 维护渠道 | 电话、邮件、现场支持，详细分级与时限见第 9 章 |
| 版本管理 | 维护期内仅发布缺陷修复版本，版本号规则 `主版本.次版本.修订号-补丁号` |
| 文档与代码归档 | 设计文档、源代码、缺陷库、变更记录在乙方按规范归档，甲方有权检查 |

## 6.5 测试性

| 需求条目 | 设计措施 |
|---|---|
| 合格性测试在指定系统上进行 | 测试主机统一使用银河麒麟 V10 上位机，环境配置随测试方案归档 |
| 测试类型 | 功能、性能、边界、安全性、接口五类，按下表覆盖 |
| 覆盖率 | 合格性测试依据技术协议 100% 符合；每个软件特性至少 1 个正常用例；合法 / 非法边界各至少 1 个用例 |
| 不具备条件的测试项 | 通过仿真器或桩模拟硬件、断网模拟超时；在测试方案中说明验证时机与方法 |

测试覆盖矩阵：

| 模块 | 功能 | 性能 | 边界 | 安全性 | 接口 |
|---|---|---|---|---|---|
| 任务管理 | ✓ | ✓ | ✓ | ✓ | ✓ |
| 数据处理 | ✓ | ✓ | ✓ | ✓ | — |
| 硬件交互 | ✓ | ✓ | ✓ | ✓ | ✓ |
| 结果评估 | ✓ | ✓ | ✓ | ✓ | ✓ |
| 系统管理 | ✓ | ✓ | ✓ | ✓ | — |

单元测试使用 QtTest 框架，按模块组织 `test_*.cpp`；集成测试由测试组按测试用例手工执行，硬件不到位的项目使用仿真器代替。

## 6.6 环境适应性

系统需求对环境适应性无特殊要求。系统部署于银河麒麟 V10 上位机环境，未引入外部环境约束。
