# 技术方案章节质量验收清单

**定位：** writer 自检 + reviewer 审查的诊断工具集。最高原则：**技术应答即得分，得分即合格。**

> 只检查技术内容。本系统不写商务/报价/资质业绩，相关项不在检查范围。

---

## 最高原则

> **技术方案章节质量标准只有一个：能不能拿到对应技术评分点的分。**

下面所有规则是辅助判断的工具，不是终审。最终是否归档由作者判断。

---

## 一、技术应答完整性（最高优先级）

- [ ] 本章承接的每个技术评分点都有 `【应答技术评分点 X.Y】` 标记的实质应答
- [ ] 应答内容对应招标"评分表-技术部分"的应答要求，不偏题
- [ ] 客观技术分应答有举证（技术指标/检测/设计值）
- [ ] 主观技术分应答有论证（结论→量化→图表→技术细节）
- [ ] 无遗漏技术评分点（逐条对照 scoring-checklist.md）

## 二、实质性技术要求规避

- [ ] 实质性★技术指标无负偏离
- [ ] 无低于技术任务书硬性门槛的参数
- [ ] 强制技术标准/规范/认证已响应或显式占位
- [ ] 无任何触碰 disqualification.md（实质性技术否决项）的表述

## 三、技术可核验性（反空话）

- [ ] 用技术指标/数据/表格/流程/清单，而非形容词
- [ ] 技术承诺有实现方式与统计口径（如可用率有技术实现）
- [ ] 无空喊口号："采用先进技术""高度可靠""国内领先"等无信息量表述
- [ ] 技术参数真实或占位标注 `【待提供：...】`，无臆造

## 四、AI 味检测（详见 anti-ai.md）

- [ ] 疲劳词未超阈值（"充分""有效""不断""持续""全面"等）
- [ ] 无连续同句式堆砌
- [ ] 无"综上所述""总而言之"类议论文腔（除非技术小结需要）
- [ ] 句式克制，技术文风，术语准确

## 五、技术方案类型专属产物

- [ ] 软件开发：技术架构、功能模块设计、接口/数据库设计、性能/安全设计
- [ ] 硬件开发：硬件设计、关键技术指标响应、技术参数偏离表、可靠性/EMC 设计
- [ ] 系统集成：总体集成架构、选型清单、技术参数偏离表、兼容性/迁移割接方案
- [ ] 服务：服务技术体系、技术服务流程、可量化技术能力指标、应急/容灾技术方案

## 六、一致性

- [ ] 技术数据/术语/承诺与其他章节、与 architecture/ 一致
- [ ] 引用检索素材已去敏、已标注来源（仅技术素材）

## 七、技术质量

- [ ] 表格优先于冗长段落
- [ ] 章节结构清晰（标题层级规范）
- [ ] 图表可解析（Mermaid 语法正确）

---

## 验收报告格式

```markdown
## 技术方案章节验收报告：第 X 章

### 总评
✅ 通过 / ⚠️ 有条件通过 / ❌ 需修改

### 技术应答覆盖
| 技术评分点 | 应答 | 质量 |
|-----------|------|------|
| 3.1 | ✅ | 优 |
| 3.2 | ❌ 缺失 | — |

### 未通过项
| 检查项 | 结果 | 证据 | 整改 |
|--------|------|------|------|
```
