# 需求分析报告格式规范

`analysis/requirement-analysis.md` 的标准结构。analyst 产出，bid-agent/architect/writer 全程参考。

## 标准结构

```markdown
# 招标需求分析报告

## 一、关键信息
| 字段 | 内容 | 原文出处 |
|------|------|---------|
| 招标编号 | ... | 封面/须知 |
| 项目名称 | ... | ... |
| 采购人 / 招标代理 | ... | ... |
| 预算 / 最高限价 | ... | 须知 X.X |
| 标段划分 | ... | ... |
| 投标截止 / 开标时间 | ... | ... |
| 投标保证金 | 金额/形式/截止 | ... |
| 投标有效期 | ... 天 | ... |
| 资格要求 | 资质/业绩/人员 | ... |

## 二、评分办法概览
- 评分总分：100
- 分项构成：技术 X / 商务 Y / 价格 Z / 资质业绩 W
- 评标方法：综合评分法 / 最低价法 / ...
- 客观分 / 主观分占比

## 三、投标可行性
- 资格门槛满足情况（逐条，标注需作者确认项）
- 可投标结论 / 风险提示

## 四、得分策略
- 客观分必拿项清单
- 主观分发力点（差异化亮点方向）
- 低性价比可放弃项（如有）

## 五、重点关注与风险
- 易废标点（指向 disqualification.md）
- 易漏项
- 模糊条款（建议澄清的问题清单）

## 六、技术要求摘要（交付 architect）
- 硬性约束（性能/容量/安全等级/合规/技术栈）
- 招标技术偏好（信创/等保/双活等）
```

## 原则

- 关键数字加粗，每项附原文出处
- 模糊/缺失项显式标注"原文未明确"，不臆造
- 第六节是 architect 的输入，技术约束要可执行
