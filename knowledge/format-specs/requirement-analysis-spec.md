# 技术需求分析报告格式规范

`analysis/requirement-analysis.md` 的标准结构。analyst 产出，bid-agent/architect/writer 全程参考。

> **范围红线：只分析技术。** 信息来源仅限招标文件的：**评分表-技术部分**、**实质性要求-技术产品部分**、**技术任务书 / 采购需求技术部分**。不分析商务、报价、付款、资质业绩等非技术内容。

## 标准结构

```markdown
# 技术需求分析报告

## 一、项目技术概况
| 字段 | 内容 | 原文出处 |
|------|------|---------|
| 招标编号 | ... | 封面/须知 |
| 项目名称 | ... | ... |
| 技术方案类型 | 软件开发/硬件开发/系统集成/服务 | 据采购需求判断 |
| 标段（技术相关）| ... | ... |
| 技术任务书/采购需求技术部分位置 | 章节/页 | ... |

## 二、技术评分办法（取自评分表-技术部分）
- 技术部分总分：X（仅技术，不含商务/价格分）
- 技术评分项构成：方案 / 产品参数 / 实施 / 服务能力 等
- 客观技术分 / 主观技术分 占比

## 三、实质性技术要求（取自实质性要求-技术产品部分）
- 实质性★技术指标清单（负偏离即否决）
- 强制技术标准/规范/认证要求
- 必须提供的技术文件/技术承诺

## 四、技术任务书要点（取自技术任务书/采购需求技术部分）
- 功能/性能/接口/安全/可靠性等技术要求摘要
- 技术难点与重点

## 五、技术得分策略
- 客观技术分必拿项
- 主观技术分发力点（技术先进性/差异化）

## 六、技术风险与关注
- 易否决的实质性技术点（指向 disqualification.md）
- 易漏的技术评分点
- 模糊技术条款（建议技术澄清）

## 七、交付 architect 的技术约束
- 硬性技术约束（性能/容量/安全等级/合规/技术栈/国产化）
- 招标技术偏好
```

## 原则

- 关键技术指标加粗，每项附原文出处
- 只取技术相关三类来源；遇到商务/价格/资质内容一律不纳入
- 模糊/缺失技术条款显式标注"原文未明确"，不臆造
- 第七节是 architect 的输入，技术约束要可执行
