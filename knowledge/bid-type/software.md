# 软件开发类技术方案撰写要点

适用：应用系统/平台/软件产品的定制开发、二次开发类项目的**技术方案**编写。

> 编写依据只取招标文件的：**评分表-技术部分**、**实质性要求-技术产品部分**、**技术任务书 / 采购需求技术部分**。不涉及商务、报价、资质业绩等内容。

## 核心得分逻辑

技术评委关注：**需求理解是否到位、技术方案是否先进可行、设计是否完整、能否落地交付。** 一切应答对应"评分表-技术部分"的每个技术评分点。

## 必备章节与侧重

### 1. 项目需求理解与技术目标
- 对技术任务书/采购需求技术部分的逐项理解
- 技术难点与重点分析、技术目标

### 2. 总体技术方案与技术架构
- 总体技术路线、技术栈选型（语言/框架/中间件/数据库）及理由
- 系统总体架构（分层/微服务），引用 architecture/ 图

### 3. 功能模块设计
- 功能模块划分与模块关系（功能模块图）
- 各模块功能点对应技术任务书需求

### 4. 详细技术设计
- 接口设计（API/协议/数据交换）
- 数据库设计（模型/性能/容量）
- 关键算法/关键技术实现

### 5. 非功能设计
- 性能（并发/响应/吞吐指标）、可扩展性
- 安全设计（认证授权/数据加密/等保合规）
- 高可用/容灾

### 6. 开发与质量保障（技术侧）
- 开发流程（敏捷/迭代）、代码质量、CI/CD
- 测试方案（单元/集成/性能/安全测试）

### 7. 部署与实施（技术侧）
- 部署架构、环境规划、数据迁移技术方案、上线技术保障

## 常见失分点

- 需求理解泛泛、未对应技术任务书逐项 → 主观分掉档
- 技术选型无理由、与需求不匹配
- 只有功能罗列，缺接口/数据/非功能设计深度
- 安全/性能仅口号无量化指标

## 实质性要求（技术产品部分）红线

- 实质性★技术指标负偏离 → 否决
- 强制技术规范/标准（如等保、国产化、特定协议）未响应 → 否决
- 必须提供的技术承诺/技术文件缺失 → 否决
