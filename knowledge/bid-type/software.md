# 软件开发类技术方案撰写要点

适用：应用系统/平台/软件产品的定制开发、二次开发类项目的**技术方案**编写。

> 编写依据只取招标文件的：**评分表-技术部分**、**实质性要求-技术产品部分**、**技术任务书 / 采购需求技术部分**。不涉及商务、报价、资质业绩等内容。

## 核心得分逻辑

技术评委关注：**需求理解是否到位、技术方案是否先进可行、设计是否完整、能否落地交付。** 一切应答对应"评分表-技术部分"的每个技术评分点。

## 强制完整性（缺项即失分/视为设计不完整）

软件类技术方案**必须**完整包含以下三大块，逐项不可缺：

1. **系统总体技术架构** —— 必须包含：**逻辑分层、数据主流向、组件图、部署图**
2. **每个功能模块的设计**（逐模块）—— 每个模块必须包含：**功能模块描述、操作步骤、类与算法设计、用例描述、界面设计**
3. **数据库设计 与 接口设计**

## 必备章节与侧重

### 1. 项目需求理解与技术目标
- 对技术任务书/采购需求技术部分的逐项理解
- 技术难点与重点分析、技术目标

### 2. 系统总体技术架构（四项必含，缺一不可）
- **逻辑分层**：表现层/应用层/服务层/数据层等逻辑分层及各层职责（配分层架构图）
- **数据主流向**：核心业务数据从产生→处理→存储→消费的主流向（配数据流向图 / 数据主流向说明）
- **组件图**：系统主要组件（服务/模块/中间件）及组件间依赖与调用关系（配组件图）
- **部署图**：物理/云部署拓扑——节点、网络、中间件、数据库的部署分布与高可用（配部署图）
- 配套：总体技术路线、技术栈选型（语言/框架/中间件/数据库）及理由

> 四张图均以 Mermaid 出图，规范见 diagram-spec.md（逻辑分层图 / 数据主流向图 / 组件图 / 部署图）。

### 3. 功能模块设计（**逐个功能模块**，每个模块五要素必含）
对技术任务书要求的**每一个功能模块**，逐模块给出：
- **功能模块描述**：模块目标、职责范围、与其他模块关系、对应的技术任务书需求点
- **操作步骤**：用户/系统的主要操作流程（分步骤，可配流程图/时序图）
- **类与算法设计**：核心类/对象设计（类图或类职责说明）+ 关键算法逻辑（流程/伪代码/复杂度）
- **用例描述**：典型用例（参与者、前置条件、主流程、备选流程、后置条件）
- **界面设计**：关键界面布局与交互说明（界面草图/线框/字段与控件说明）

> 建议每个模块用统一小节模板（见 bid-format-spec.md "功能模块设计模板"），确保五要素齐全、便于评委逐模块核对。

### 4. 数据库设计
- 概念/逻辑/物理模型（E-R 或表结构）、关键表与字段、主外键与索引
- 数据容量/性能/分库分表、备份与一致性

### 5. 接口设计
- 接口清单（内部/外部）、协议（REST/gRPC/MQ 等）、请求/响应规范、鉴权、错误码
- 数据交换格式、幂等/限流/版本管理

### 6. 非功能设计
- 性能（并发/响应/吞吐指标）、可扩展性
- 安全设计（认证授权/数据加密/等保合规）
- 高可用/容灾

### 7. 开发与质量保障（技术侧）
- 开发流程（敏捷/迭代）、代码质量、CI/CD
- 测试方案（单元/集成/性能/安全测试）

### 8. 部署与实施（技术侧）
- 部署架构（与第 2 节部署图一致）、环境规划、数据迁移技术方案、上线技术保障

## 常见失分点

- 总体架构缺四项之一（逻辑分层/数据主流向/组件图/部署图）→ 设计不完整、主观分掉档
- 功能模块只给"模块列表"，未逐模块给五要素 → 视为设计深度不足
- 用例/界面设计缺失（评委无法判断可落地性）
- 需求理解泛泛、技术选型无理由
- 安全/性能仅口号无量化指标

## 实质性要求（技术产品部分）红线

- 实质性★技术指标负偏离 → 否决
- 强制技术规范/标准（如等保、国产化、特定协议）未响应 → 否决
- 必须提供的技术承诺/技术文件缺失 → 否决
